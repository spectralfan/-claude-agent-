## 一、项目简介

**项目名称**：JChatMind — 多智能体协作 AI 编码平台

**一句话**：基于 Spring Boot 3.5 + Spring AI 1.1 构建的多智能体 AI 编码平台，支持 Scheduler-Worker-Reviewer 三层 DAG 编排、MCP 安全执行链与分层记忆系统，实现类 Claude Code 的自主开发能力。

**技术栈**：

| 层级 | 技术 |
|------|------|
| 后端 | Java 17, Spring Boot 3.5.8, Spring AI 1.1.0, MyBatis 3.0.3 |
| 数据库 | PostgreSQL (pgvector, JSONB), 乐观锁并发控制 |
| 前端 | React 19, TypeScript, Vite 7, Ant Design X 2.0, Tailwind 4 |
| LLM | DeepSeek, 智谱 GLM-4 |
| 工具 | MCP (SSE + STDIO), Ollama (bge-m3 向量嵌入), Git |
| 消息 | SSE 实时推送, RocketMQ (可选) |

**核心能力标签**：
- 多 Agent 协作：Scheduler 拆解需求 → Worker 并行编码 → Reviewer 自动审查，全流程自动闭环
- 安全工具执行：三层 MCP 安全防线 + 结构化验证优先策略，杜绝 Agent 自主生成高危命令
- 分层记忆系统：WORKING/RECENT/ARCHIVE 三级记忆，结合向量检索与重要性评分
- 实时可见性：统一 SSE 协议覆盖 Agent 闭环与 Coding 全生命周期

## 二、简历正文（项目经历用）

### 项目简介

**JChatMind** — 基于 Spring Boot 3.5 + Spring AI 1.1 的多智能体协作 AI 编码平台，实现自主决策、多 Agent 协作编码、安全工具执行与分层记忆检索，支持 DeepSeek/智谱 GLM 多模型切换。

**技术栈**：Java 17 / Spring Boot 3.5 / Spring AI 1.1 / PostgreSQL pgvector / MyBatis / React 19 / Ant Design X / DeepSeek & GLM-4

---

### 技术亮点

1. **ReAct Agent 闭环引擎**：设计 PLANNING→THINKING→EXECUTING→OBSERVING 显式状态机，手动接管 Spring AI 工具执行控制，支持多轮工具调用与任务规划。按 tool 调用轮次裁剪上下文（ToolAwareMessageWindowChatMemory），钉住 system + 首条 user 消息，避免深循环下 tool 链截断导致 LLM 400 错误；sanitizeMessagesForToolCallingApi 作为最后防线清理悬空 tool 消息。EventBus + AgentLoop 通用引擎提供会话级乐观锁并发控制与 Run 维度生命周期管理。

2. **工具体系 + MCP 全链路执行链**：设计 FIXED/OPTIONAL/Coding/编排 四类共 20+ 本地工具（文件/Git/URL抓取/结构化验证/Shell）灵活配置框架。MCP 执行链包含：McpProxyApplicationContextInitializer 在 Spring 容器前拉起 mcp-proxy（避免首次 ConnectException）；McpToolAliasRegistry 将 run_terminal_cmd/shell_exec/execute_command 等 6 种命名统一映射到 bash；三层安全防线结构化验证优先（绕过 MCP 拼 shell）→ 策略门禁（McpShellCommandPolicy 拦截 node -e 等 12+ 高危模式）→ 参数改写（McpShellArgumentEnricher 路径安全化 + command-runner.mjs 跨平台执行）；RecordingToolCallback 异步写入调用埋点表，CodingMcpOutputBridge 将输出转为 SSE 推送到前端终端面板。

3. **Memory Hub 分层记忆系统**：设计 WORKING/RECENT/ARCHIVE 三级记忆架构。WORKING 短期滑动窗口镜像当前会话消息；RECENT 中期通过重要性评分 Top-N 筛选跨会话高价值记忆；ARCHIVE 长期基于 Ollama bge-m3 语义嵌入 + pgvector 索引实现向量检索持久化。Agent 加载历史时通过 ToolResultCompactor 对较早工具调用结果按策略压缩——验证类仅保留退出码、Shell 类保留首尾各 800 字符、读文件类截断至 2000 字符，最近 2 轮保持完整；压缩后的上下文镜像到 Memory Hub。每条消息自动镜像到 WORKING 层，MemoryAgent 异步整理并调整层级，Coding 会话自动注入 RECENT/ARCHIVE 记忆恢复长期上下文。

4. **Session 文件持久化与生命周期管理**：每个会话在磁盘上维护三层文件——thread.jsonl（追加写 JSONL，记录完整对话消息流，是消息的权威源，DB 仅作查询索引）、notes.md（Agent 自主记录的事实笔记，Markdown 格式人类可读）、meta.json（状态/版本/run 计数）。SessionManagerImpl 统一管理 CREATED→ACTIVE⇄PAUSED→COMPLETED/FAILED 状态机生命周期，version 字段乐观锁防并发冲突。前端路由根据 CHAT/CODING 类型化分离入口，后端按类型加载不同工具与上下文。

5. **SSE + EventBus 实时事件推送**：设计 12+ 事件类型的统一 SSE 协议，Agent 闭环五阶段（AI_PLANNING/THINKING/EXECUTING/OBSERVING/DONE）与 Coding 全生命周期（CODING_STARTED/FILE_CHANGED/COMMAND_OUTPUT/SUBTASK_*/COMPLETED/FAILED）全覆盖。ChatEventPublisher 接口实现可拔插——LocalChatEventPublisher 本地直推 SSE、RocketMqChatEventPublisher 生产部署桥接。RealtimeNotifier.tryPublish() 统一入口，前端 useSessionSse hook 单连接订阅所有事件类型。

6. **多 Agent DAG 编排**：设计 Scheduler/Worker/Reviewer 三角色体系，CodingAgentRoles 根据工具清单自动判定角色、按白名单做工具级隔离。Scheduler 将需求拆解为有依赖关系的 DAG → 多个 Worker 按 dependsOn 解锁后 @Async 并行编码 → Reviewer 自动审查产出 → OrchestratorContinuationService 全部终态后自动续跑 Scheduler 进入下一轮。基于 PostgreSQL t_orchestration_task 表驱动状态机（PENDING→READY→RUNNING→COMPLETED/FAILED），max-concurrency 控制并发度。SubAgentExecutor 独立子会话零父历史 prompt，避免上下文污染。

### 项目难点

**1. Agent 闭环稳定性与 tool 链断裂**
Agent 在长循环场景下多次调用工具后，普通滑动窗口会裁断 tool_calls 与 tool 响应的配对关系，DeepSeek/OpenAI 严格要求两者成对出现，否则返回 400 错误。设计 ToolAwareMessageWindowChatMemory 按 tool 调用轮次整轮淘汰、钉住 system + 首条 user 消息，配合 sanitizeMessagesForToolCallingApi 清理悬空 tool 消息，从根源解决 tool 链断裂问题。

**2. MCP 工具安全管控**
Agent 自主编码时可能生成 node -e 执行任意代码、http-server 启动 HTTP 服务、重定向管道等危险命令。设计三层递进安全防线：结构化验证优先路径（CodingVerifyTools 不经 MCP 拼 shell）→ 策略门禁（McpShellCommandPolicy 正则拦截 12+ 高危模式）→ 参数改写与安全执行（McpShellArgumentEnricher 路径相对化 + command-runner.mjs 跨平台执行），确保 Agent 只能执行预期内的操作。

**3. 长任务上下文管理**
Coding 任务 Agent 可能执行 100+ 轮工具调用，消息量远超普通对话窗口。设计消息窗口取 max(agent配置, 80) 条，启用 ToolAware 按轮裁剪；ToolResultCompactor 对较早工具轮次按策略压缩（状态摘要/首尾保留/字符截断），最近 N 轮保持完整；Memory Hub 的 RECENT/ARCHIVE 层通过重要性评分和向量检索恢复长期上下文，避免关键决策丢失。

**4. 多 Agent 协作上下文隔离**
子 Agent（Worker/Reviewer）如果继承父会话完整历史，不仅 token 浪费严重，还会造成上下文污染——子 Agent 看到不相关的父会话消息可能误判。设计 SubAgentExecutor 为每个子任务创建独立子会话（零父历史 prompt），仅注入角色模板 + 任务目标 + 上下文文件，从架构上保证子 Agent 上下文纯净。

**5. 自动续跑循环失控**
Worker 完成编码后自动触发 Scheduler 下一轮，如果 Scheduler 判断条件过于宽松可能形成无限循环。设计 OrchestratorContinuationService 在每次续跑前检测 DAG 终态：所有节点 COMPLETED/FAILED 且无新增任务时停止；配合 max-round 上限（当前 5 轮）作为硬性兜底，超过则标记会话完成。

**6. Agent 可观测性设计**
Agent 闭环（PLANNING→THINKING→EXECUTING→OBSERVING）的状态转换、工具调用参数及结果、MCP 命令执行输出对前端完全不可见，用户只能看到最终结果，无法理解 Agent 决策过程。统一 SSE 协议将 12+ 事件类型分阶段推送；RecordingToolCallback 将每次 MCP 工具调用异步写入 t_mcp_tool_call 表；前端 CodingView 同时渲染 AI 对话、终端面板、文件变更 Diff 和子任务状态树，实现 Agent 执行过程完全透明。

## 三、面试话术速查表

| 面试问题 | 回答要点 |
|---------|---------|
| 和普通 Agent 框架有什么区别？ | 手动 ReAct 闭环 + 多 Agent DAG 编排 + MCP 三层安全 + 分层记忆，不只是对话，而是自主编码平台 |
| 为什么不用 LangChain？ | Spring AI 与 Spring Boot 生态深度集成，自定义 ReAct 闭环更可控，避免框架黑盒 |
| Tool 链断裂怎么解决的？ | ToolAwareMemory 按轮裁剪 + 钉住 system/user + sanitizeMessages 最后一道防线清理悬空消息 |
| MCP 安全架构怎么设计的？ | 三层递进：结构化验证优先（不经 MCP）→ 策略门禁拦截高危命令 → 参数改写 + 跨平台安全执行 |
| 长任务上下文怎么管理的？ | max(80条)窗口 + ToolAware 按轮裁剪 + ToolResultCompactor 压缩 + Memory Hub 向量检索恢复 |
| 子 Agent 上下文污染怎么避免？ | SubAgentExecutor 创建独立子会话，仅注入角色模板 + 目标 + 文件，零父历史 prompt |
| DAG 自动续跑怎么防无限循环？ | OrchestratorContinuationService 终态检测（所有节点终态 + 无新增任务）+ max-round=5 硬兜底 |
| 分层记忆和简单 RAG 的区别？ | RAG 只做语义搜索；分层记忆加入重要性评分、三级自动流转、MemoryAgent 异步整理 |
| Proxy 为什么要在容器前启动？ | ApplicationContextInitializer 在 Spring 刷新前拉起 proxy，避免 Agent 首次工具调用 ConnectException |
