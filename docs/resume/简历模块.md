## 二、简历正文（项目经历用）

### 项目简介

**JChatMind** — 基于 Spring Boot 3.5 + Spring AI 1.1 的多 Agent 协作 AI 编码平台。Agent 通过 ReAct（Reason→Act→Observe）闭环自主决策，调用内置 BashTool/MCP 工具、委派子 Agent 协作完成任务，配合三层记忆与事件总线实现端到端编码。

**技术栈**：Java 17 / Spring Boot 3.5 / Spring AI 1.1 / PostgreSQL pgvector / MyBatis / React 19 / TypeScript / Ant Design X / DeepSeek & GLM-4

---

### 技术亮点

1. **ReAct Agent 闭环引擎**：设计 PLAN→THINK→ACT→OBSERVE 显式状态机，LLM 在每轮自主决策工具调用。手动接管 Spring AI 工具执行控制，支持单轮多工具并行调用。EventBus 内存总线实时发布 12+ 事件类型，EventWriter 持久化 events.jsonl 支持断线回放。按 tool 轮次裁剪上下文（ToolAwareMessageWindowChatMemory），钉住 system + 首条 user 消息，避免深循环下 tool 链断裂。

2. **子 Agent 委派与轻量任务追踪**：通过 spawn_agent 工具按需创建 planner/worker/reviewer 子 Agent，每个角色由独立的 YAML Profile 配置 systemPrompt 和工具白名单。子 Agent 深度控制（子可再 spawn 孙），执行过程通过 EventBus 桥接回父 Agent。配合 task_create/update/list/get 四工具，LLM 自主将需求拆解为子任务并追踪进度（pending→in_progress→completed），支持 blocked_by 依赖自动维护。

3. **双通道工具执行体系**：自建 BashTool（Java ProcessBuilder 直连，自动注入 coding workspace 为工作目录，64KB 输出截断，120s 超时）处理高频率 shell 命令；MCP 协议接入外部扩展工具。ToolRegistry 统一注册两种执行通道，ToolFacadeService 按 FIXED/OPTIONAL 类型和 Agent 白名单灵活过滤。RecordingToolCallback 埋点记录每次调用。

4. **分层记忆与上下文压缩系统**：三层记忆架构——WORKING（滑动窗口 20 条）、RECENT（重要性≥7 自动升级）、ARCHIVE（PgVector 语义检索）。四层压缩协同运转：ToolResultCompactor 实时裁剪 bash/list 等大输出；MemoryAgent 会话结束后用 LLM 将 RECENT 浓缩为中文摘要写入 ARCHIVE 并向量化；MemorySelector 按 token 预算融合三层注入。MemoryIntegration 自动保存工具调用和用户确认。

5. **6 层权限评估体系**：设计 deny_patterns → outside_cwd 检测（/abs、~、..、$HOME、cd 等触发强制审批）→ session 缓存 → persistent 缓存（policy.json 持久化）→ allow_patterns → default 策略的 6 层静态评估。写文件/执行命令走 Future 异步审批，前端弹窗等待用户决策，60s 超时自动拒绝。支持 allow_once/always_allow/deny_once/always_deny 四种决策。setMode("auto") 自动放行。

6. **EventBus 事件总线 + 实时通信**：CopyOnWrite 内存事件总线作为中枢，Agent 执行期间所有事件同步分发两条通路——RpcEventBridge 推送 WebSocket 实时渲染前端，EventWriter 以 NDJSON 持久化到 runs/{runId}/events.jsonl。前端断线重连时通过 ReplayController（REST API）或 WebSocket replay_run_id 参数自动回放。子 Agent 进度独立渲染，权限审批弹窗走独立通道。

### 项目难点

**1. Agent 工具链稳定性**
Agent 在长循环场景下多次调用工具后，普通滑动窗口会裁断 tool_calls 与 tool 响应的配对关系，LLM 严格要求两者成对出现，否则返回 400 错误。设计 ToolAwareMessageWindowChatMemory 按 tool 调用轮次整轮淘汰、钉住 system + 首条 user 消息，配合 sanitizeMessagesForToolCallingApi 清理悬空 tool 消息，从根源解决 tool 链断裂问题。

**2. 工具执行目录错乱**
Agent 同时操作文件工具（write_coding_file）和 shell 命令（execute_command）时，文件工具通过 CodingWorkspaceService 解析路径到 coding workspace，但 shell 命令的 working directory 却是 Java 进程启动目录（项目根），导致 Agent 在错误目录操作。将基于 MCP proxy 的 shell 执行替换为自建 BashTool（ProcessBuilder 直连），通过 `pb.directory(workspace)` 系统调用设置工作目录，解决路径不统一问题。

**3. 子 Agent 路径重复**
Agent 在 spawn 子 Agent 时，prompt 中附带相对路径（如 ./test-workspace），但子 Agent 的工作区已经锁定到该目录，导致文件写入路径叠加（workspace/test-workspace/test-workspace/）。在 JChatMindFactory.createProfileSubAgent 中为 worker 子 Agent 的 system prompt 追加当前工作区路径声明，消除 Agent 对工作目录的误判。

**4. 权限审批前后端断连**
前端审批弹窗通过 WebSocket 收到 PermissionRequestedEvent 后调用 http.post 发送决策，但 http 模块内部已在 URL 前拼接 /api 前缀，前端又硬编码了完整 BASE_URL（http://localhost:8080），导致实际请求路径为 /apihttp://localhost:8080/... 的垃圾 URL，后端从未收到 respond 请求，Future 永远阻塞直到 60s 超时拒绝。去掉 BASE_URL 硬编码，使用 http.post('/mcp/permission/respond') 相对路径解决。

**5. 文件系统任务状态过期**
Agent 在 coding workspace 下创建 .tasks/task_*.json 记录子任务状态。当用户新建 session 或切换 workspace 时，旧 .tasks 文件仍存在，Agent 读到过期状态认为任务已完成。在 CodingTaskAutoProvisionerImpl 中增加 workspace 变更检测和 .tasks 自动清理，匹配旧任务时重置状态。

**6. Agent 可观测性**
Agent 闭环的状态转换、工具调用参数及结果对前端完全不可见，用户只能看到最终结果。设计 EventBus 同步分发所有事件到 RpcEventBridge（WebSocket 推送）和 EventWriter（NDJSON 持久化）；前端断线重写后通过 ReplayController 回放完整执行过程；子 Agent 进度独立渲染，权限审批弹窗走独立事件通道。
